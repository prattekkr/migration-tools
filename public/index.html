            XMOD S7 Package Updater
            ZIP → ZIP rewrite
          

          

            

              Post-processes a downloaded content-package ZIP: swaps DAM paths back to original asset URLs
              (via asset-mapping.json) and/or relocates the JCR content to a new target path —
              auto-detecting the source path from META-INF/vault/filter.xml.
            

            

              

                
Steps

                
    
                  
  1. Upload: Select a bundle ZIP (with asset-mapping.json + inner content zip) or a plain content ZIP. Optionally attach an asset-mapping.json for a plain ZIP.
    2. 
                  
  2. Inspect: See whether it's a bundle, the detected source path, and how many asset replacements will run.
    3. 
                  
  3. Target (optional): Enter a target JCR path to move the content; leave blank to only replace asset references.
    4. 
                  
  4. Process: Download the updated ZIP. The change log lists every modified, sanitized, and created entry.
    5. 
                

              

              

                
Notes

                
    
                  
  • Asset replacement maps damPath → url (the inverse of the Image/Asset tool) and runs for bundles or an attached mapping.
    • 
                  
  • Path move renames jcr_root{source} → jcr_root{target}, rewrites path strings in files (incl. filter.xml), and creates intermediate cq:Page nodes.
    • 
                  
  • Binary assets are copied untouched; bare & in XML is escaped to &amp;.
    • 
                  
  • This tool intentionally edits package metadata when moving — that's required for the new filter roots.
    • 
                

              

            

          

        

      


      
      

        

          

            
            Page Metadata Migration
            AEM → EDS
          

          

            

              Reads page properties from AEM author and writes the mapped values into Edge Delivery Services (EDS)
              metadata sheets. Useful for bulk-migrating titles, descriptions, tags, and any custom properties
              without touching each page manually.
            


            

              

                
Steps

                
    
                  
  1. Step 1 — Configure: Enter your AEM Author URL + credentials and the target EDS SharePoint / Drive URL. Click Save Config, then Test Connections to verify both are reachable.
    2. 
                  
  2. Step 2 — Discover & Map: Enter one or more AEM root paths (e.g. /content/abbvie-com) and click Discover Pages. The tool streams all pages via AEM QueryBuilder. Once done, map each AEM property to its EDS column name. Optionally apply a transform (uppercase, lowercase, date format, etc.).
    3. 
                  
  3. Step 3 — Bulk Update: Review the discovered pages, check / uncheck individual pages, then click Start Bulk Update. Progress is streamed live. Download the CSV or log when finished.
    4. 
                

              

              

                
Tips & Notes

                
    
                  
  • Credentials are saved to config.json on the server — no need to re-enter each session.
    • 
                  
  • Property mappings are saved to mapping.json and persist across sessions.
    • 
                  
  • Use Preview Update on a single page before running the full bulk update.
    • 
                  
  • AEM SSL errors are suppressed automatically (rejectUnauthorized: false).
    • 
                  
  • The EDS URL must point to the metadata sheet of your SharePoint workbook.
    • 
                

              

            

          

        

      


      
      

        

          

            
            Image / Asset Migration
            DAM → Dynamic Media
          

          

            

              Rewrites AEM DAM asset URLs inside an EDS HTML/JSON ZIP package to Dynamic Media Open API (DM) URLs.
              Supports multi-environment configs (dev / stage / prod) and custom path mappings.
            


            

              

                
Steps

                
    
                  
  1. Step 1 — Environment: Pick a mode (AEM to DM or Domain Swap) and select the target environment. Use Use AEM from Page Metadata Config to auto-fill AEM credentials.
    2. 
                  
  2. Step 2 — Asset CSV: Click Build CSV to query AEM DAM and generate a mapping of old DAM paths → new DM URLs. The status pill shows whether a CSV already exists for the environment.
    3. 
                  
  3. Step 3 — Upload ZIP: Upload your EDS output ZIP. Add root path mappings if your DAM paths differ between environments. Click Process ZIP to rewrite all asset references.
    4. 
                  
  4. Step 4 — Download: Download the patched ZIP and a change report CSV. For Domain Swap mode, upload the ZIP and enter old / new domain strings.
    5. 
                

              

              

                
Tips & Notes

                
    
                  
  • Environment configs live in site.config.json. Edit it to add new environments or change DM delivery domains.
    • 
                  
  • The asset CSV is cached per environment in data/. Rebuild it only when DAM assets change.
    • 
                  
  • Root mappings let you strip or replace a DAM path prefix before looking up the DM URL (useful when dev/stage paths differ from prod).
    • 
                  
  • Custom mappings let you hard-code specific DAM path → DM URL pairs that override the CSV.
    • 
                  
  • Build logs stream in real time. Scroll the log box to see per-file results.
    • 
                

              

            

          

        

      


      
      

        

          

            
            Link Checker
            AEM Package → Link Audit
          

          

            

              Scans every XML file inside an AEM content package ZIP and extracts all links, classifying them
              by type. Flags links that need attention — shortened paths that skip the /content/
              prefix, and absolute AbbVie URLs that should be relative.
            


            

              

                
Steps

                
    
                  
  1. Configure Site Roots: Before checking, expand Site Root Configuration and add your content roots (e.g. /content/abbvie-com/us/en). These persist in the browser and are used to suggest correct full paths.
    2. 
                  
  2. Upload ZIP: Click the upload zone and select an AEM package ZIP (supports nested outer→inner ZIP structures).
    3. 
                  
  3. Check Links: Click Check Links. The server scans all XML files and returns a full link inventory with counts per type.
    4. 
                  
  4. Review Results: Use the filter pills to focus on a specific type. Expand any file row to see individual links. Search by file name or URL.
    5. 
                  
  5. Fix Short Paths: If short paths are detected, a Fix & Download ZIP panel appears. Select a site root prefix and download the patched ZIP with all short paths corrected.
    6. 
                  
  6. Export: Click Export CSV to download the full link report (respects the current type filter).
    7. 
                

              

              

                
Link Types

                

                  
DAM Links starting with /content/dam/ — DAM asset references

                  
Internal Page links starting with /content/ (non-DAM)

                  
External Absolute https:// links to third-party domains

                  
Scene7 Classic Scene7 URLs (*.scene7.com, /is/image/)

                  
DM Open API Dynamic Media Open API — delivery-p{n}-e{n} or /adobe/assets/ URLs

                  
AEM Cloud Links pointing to adobeaemcloud.com or adobe.com

                  
AbbVie (Abs) Absolute abbvie.com / www.abbvie.com links — should be relative

                  
Short Path Paths like /who-we-are/page missing the /content/ prefix

                

                

                  
Tips

                  
    
                    
  • JCR / Sling XML namespace URIs (e.g. http://www.jcp.org/jcr/1.0) are automatically excluded.
    • 
                    
  • The fixer only patches short paths — DAM, internal, and external links are left unchanged.
    • 
                    
  • After fixing, the original session expires. Re-upload to fix again or use a different prefix.
    • 
                  

                

              

            

          

        

      


      
      

        

          

            
            Package Creator
            Package Manager API
          

          

            

              Bulk-creates an AEM content package and applies a set of workspace filter roots via the
              Package Manager API — then optionally builds it. Useful for assembling an export package
              over many content paths without hand-editing filter.xml.
            

            

              

                
Steps

                
    
                  
  1. Connection: Enter the AEM author host, username and password and click Test Connection. Use Use AEM from Image/Asset config to pre-fill the host.
    2. 
                  
  2. Package Details: Set the package name, group, version and optional description.
    3. 
                  
  3. Filter Paths: Pick the default filter mode, then paste content paths (one per line) or upload a .txt file. Each path becomes a filter root.
    4. 
                  
  4. Create: Click Create & Add Filters, or Create, Add Filters & Build to also build. Progress streams live; a link to the package opens it in Package Manager.
    5. 
                

              

              

                
Notes

                
    
                  
  • Filter modes: replace (replaces existing content), merge (adds new, keeps existing), update (only updates existing nodes).
    • 
                  
  • Only lines starting with / are treated as paths; everything else is ignored.
    • 
                  
  • Self-signed AEM SSL certificates are accepted automatically.
    • 
                  
  • The package is created under /etc/packages/{group} using the name and group you provide.
    •
